# 广告需求平台开发工作流复盘

## 背景

本项目是 Vite + React + TypeScript 的广告创意与需求管理平台。过往开发不是按单个页面随手堆功能，而是围绕“方向 -> 需求 -> 脚本 / 排期 -> 成片数据回流 -> 复盘”的业务闭环逐步推进。

这份文档沉淀目前比较有效的开发流程，后续新功能、重构或 UI 调整优先按这里执行。

## 一、开始前先对齐上下文

每轮重要开发前先读三类上下文：

1. 项目协作规范：`AGENTS.md`
2. 当前产品主线：
   - `docs/requirement-platform-execution-plan.md`
   - `docs/requirement-platform-roadmap.md`
3. 模块专项规范：
   - UI 相关先参考 `components/UiSpecification.tsx`
   - 需求脚本工作台先读 `docs/requirement-script-workbench-guidelines.md`
   - 响应式调整先看 `docs/responsive-layout-guidelines.md`

这个步骤的价值是防止功能漂移。比如“成片需求是否进入资产库”这类边界，一旦确认，就要先写入计划和文档，再落代码。

## 二、需求先收口成产品边界

有效做法：

- 先用一句话定义本轮目标。
- 明确本轮不做什么。
- 把对象关系说清楚，再开始改组件。
- 如果用户纠正产品逻辑，先更新计划和文档，再继续开发。

示例：

- 正确边界：成片表现回流到需求、版本和方向复盘。
- 错误边界：成片上传后自动沉淀进资产库。
- 修正后规则：资产库保持手动录入和人工治理，`引用成片` 与 `引用资产库` 是两个独立来源。

## 三、复杂功能先写计划

多步骤功能不要直接开改，先写计划到：

`docs/superpowers/plans/`

已有计划示例：

- `docs/superpowers/plans/2026-06-09-asset-library-type-filter.md`
- `docs/superpowers/plans/2026-06-11-data-feedback-review.md`

一份可执行计划至少包含：

- 目标
- 当前上下文
- 产品规则或边界修正
- 数据模型决策
- 文件改动范围
- 分任务实施步骤
- 验收标准
- 风险和未决问题

计划不只是文档，它是后续开发的执行脚本。实现时按任务顺序推进，避免在大组件里随手补逻辑。

## 四、实现顺序：类型先行，再接数据，再接 UI

这个项目目前最稳的开发顺序是：

1. 先补 `types.ts`
   - 定义业务对象和状态枚举。
   - 避免 UI 里先出现散乱的临时字段。

2. 再补 `services/mockData.ts`
   - 先用 mock 跑通前端闭环。
   - 给汇总逻辑单独函数，例如需求 summary、方向 summary。

3. 再接容器组件
   - 例如 `components/RequirementCenter.tsx` 负责聚合状态和跨视图联动。
   - 用 `useMemo` 做 map / summary，保证同一页面内数据来源一致。

4. 再接详情或子视图
   - 例如 `components/RequirementDetail.tsx` 展示单需求的版本级数据。
   - 详情页里新增模块要尽量独立成 section，不打断已有脚本和排期工作流。

5. 最后接数据页或辅助入口
   - 例如 `components/CreativeAnalysis.tsx` 展示方向级复盘视角。
   - 不替换已有分析页，只补业务复盘入口。

6. 同步更新 docs
   - 代码里做了产品边界变化，必须同步改路线图和执行计划。

## 五、UI 开发原则

过往比较有效的 UI 规则：

- 保持业务后台风格，不做营销页式展示。
- 主流程优先，新增信息做成轻量 badge、summary strip 或独立 section。
- 状态必须有颜色区分，但不要抢主操作。
- 详情页新增模块应放在用户决策路径附近，而不是插在主要编辑区中间。
- 表格、卡片、筛选之间保持层级清楚，避免卡片套卡片。
- 大组件里新增逻辑时，优先抽 helper / summary / map，减少 JSX 里直接写复杂计算。

近期数据回流功能的 UI 落点：

- 协同看板方向卡：显示 `数据回流 x/y`。
- 需求大表：显示上线成片、放量信号、需复盘、累计消耗。
- 需求行：显示建议动作，例如 `Scale`、`Iterate`、`Pause`、`Observe`。
- 需求详情：显示版本级消耗、CPI、IR、D7 ROAS 和结论。
- 创意分析页：补方向数据回流复盘表，不覆盖原有深度分析。

## 六、验证流程

代码变更后至少跑：

```bash
npm run lint
```

涉及构建、路由、依赖、入口文件或全局样式时，同时跑：

```bash
npm run build
```

涉及主要界面时，还要启动本地服务：

```bash
npm run dev
```

如果需要给局域网设备访问：

```bash
npm run dev -- --host 0.0.0.0
```

然后使用 Vite 输出的 `Network` 地址，例如：

```text
http://10.4.20.33:3001/
```

可视化检查至少覆盖：

- 默认入口是否正常加载。
- 新增模块是否出现在预期页面。
- 表格、badge、summary 在桌面宽度下是否挤压。
- 关键点击路径是否能进入详情。
- 浏览器 console 是否有明显错误。

## 七、工具和环境经验

这个项目历史上遇到过几类环境问题：

- 本机 `npm` / `python` 不一定可靠，必要时优先使用 Codex bundled runtime。
- npm 安装或 registry 失败时，要同时考虑网络、cache 写入目录和沙箱权限。
- 本地服务绑定局域网时用 `--host 0.0.0.0`。
- headless 浏览器或 CDP 在沙箱中可能被权限限制；截图验证失败时，要把“代码构建通过”和“截图 QA 未完成”分开说明。
- 如果启动了临时 dev server 或 Chrome headless，收尾时检查端口并清理进程。

## 八、Git 与工作树纪律

当前仓库经常处在多人或多工具改动状态。开发前后都应该看：

```bash
git status --short --branch
```

规则：

- 不回滚用户已有改动。
- 不把 unrelated diff 当作本轮成果。
- 如果本轮只改一部分文件，最终说明要区分“本轮改动”和“工作树已有改动”。
- 重要功能完成后，适合做一个可回滚 commit；但提交前要确认用户是否希望现在提交。

## 九、推荐的后续标准流程

后续每个中等以上功能，建议按这个顺序跑：

1. 读 `AGENTS.md`、路线图、相关专项规范。
2. 用一句话确认目标和不做范围。
3. 写或更新 `docs/superpowers/plans/<date>-<feature>.md`。
4. 按 `types.ts -> services/mockData.ts -> 容器组件 -> 详情/子组件 -> docs` 顺序实现。
5. 每完成一个功能包跑一次 `npm run lint`。
6. 最后跑 `npm run build`。
7. 启动本地服务做可视化检查。
8. 最终回复列出改了什么、验证结果、没完成的 QA 或风险。
9. 若产生长期规则或项目决策，更新对应文档或 Obsidian 记忆。

## 十、当前沉淀结论

这个项目最适合“产品边界先行 + 数据模型先行 + 前端 mock 闭环 + 可视化验收”的节奏。

不要把它当普通后台页面逐页开发。每次都应该问：

- 这个改动是否推进了真实业务闭环？
- 字段是否能支撑后续状态和复盘？
- 是否和需求中心、需求详情、数据页使用同一套对象关系？
- 是否把新边界写进文档，避免下一轮再走回头路？
