# 需求平台执行计划

## 设计简报

这是广告创意与需求管理平台，不做营销页式的展示，而是做高频使用的业务后台。界面优先保证信息层级清楚、操作路径短、状态可追踪，并沿用 `components/UiSpecification.tsx` 的视觉规范。

当前开发优先跑通完整工作流：

1. 方向创建。
2. 方向下创建需求。
3. 需求继承方向字段并进入对应制作类型模板。
4. 需求审核。
5. 拆岗、评估难度并手动排期。
6. 制作人员推进任务状态。
7. 成品上传并回写需求、版本和制作任务状态。
8. 成片投放数据回流到需求、版本和方向，用于复盘与迭代。

## 开发原则

- 先闭环，后丰富：每次改动都尽量让一个真实工作流更完整，而不是只做单个静态页面。
- 保持业务后台密度：页面使用浅灰底承载内容区，白色卡片只用于独立对象，避免一个板子套一个板子。
- 操作层级固定：主动作必须明显，次动作靠近对应内容，危险动作收敛并二次确认。
- 字段口径统一：方向弹窗、需求大表、需求详情、协同看板使用同一套状态和字段含义。
- 手动优先：人员、时间、工时由组长或制作负责人手动填写，系统只提供人员占用、冲突和风险提示，不做一键自动改排。
- 模板分流：视频进入视频脚本模板；图片和试玩进入各自独立模板，但同样按小版本填写。

## 当前阶段

### 阶段一：需求中心闭环

已基本完成：

- 方向卡片点击打开方向详情。
- 方向下新建需求，并按方向制作类型分流。
- 需求详情修改状态后同步回协同看板。
- 视频、图片、试玩进入不同需求脚本界面。
- 需求大表默认以“新建需求”为最高优先级入口，筛选后显示列表。

继续补齐：

- 方向详情弹窗与需求大表字段继续保持一致。
- 需求编号、状态、进度按真实关联需求计算。
- 需求详情保存、删除和回写状态持续稳定。

### 阶段二：制作任务与排期

当前优先推进：

- 需求创建时自动生成默认制作任务。
- 任务支持岗位、前置关系、人员、开始时间、结束时间、预计工时。
- 人员、时间、工时由人工填写，系统只提示占用、冲突和截止风险。
- 任务状态驱动需求制作状态。
- 制作排期看板按任务粒度展示日历和甘特图，兼容旧需求级排期数据。
- 当前排期页聚焦手动排期辅助：查看人员占用、未来 7 天负荷和高优先级风险；人员排期情况通过弹窗查看，不在页面内铺开大日历。
- 需求详情手动排期时，选择负责人后可直接看到该人员未来两周占用、撞期任务和可参考空档。
- 需求详情手动排期时，新增岗位会根据需求难度和岗位类型默认填入建议工时，并在每个任务下方给出可点击的推荐人员/推荐时间。
- 制作排期页的最高/高优风险已升级为可操作列表，展示风险原因、截止时间、当前排期结束时间和建议动作。
- 协同看板方向卡片会回显同一套排期风险，让创意和组长不进入排期页也能看到高优需求的排期问题。
- 方向详情弹窗和需求大表会在需求名称旁显示同一套排期风险徽标和建议动作，帮助用户在列表层判断优先处理项。
- 需求大表支持按排期风险筛选，可查看全部、有风险或严重风险的需求。
- 排期风险已接入排序：协同看板可按方向风险排序，需求大表默认优先露出风险需求，方向详情弹窗也优先展示高风险需求。
- 制作排期页已补齐岗位 / 人员产能视图，按在职人员分组显示未来 7 天占用、任务数、最近空档，并可跳转到人员排期弹窗。

下一步：

- 继续打磨需求详情手动排期区的冲突提示和截止风险提示，使选择人员 / 时间时能即时反馈。

### 阶段三：手动排期体验增强

先做前端可验证版本：

- 按岗位给出推荐人员分组。
- 选择人员后显示未来占用、冲突任务和可参考空档。
- 输入时间后提示是否撞期或超过方向截止日期。
- 输入人员后在看板中定位该人员当前排期。

需要暂停确认的决策：

- 飞书请假同步方式和权限。
- 中国节假日表的维护来源。
- 不同岗位的默认承接优先级和人员名单。

### 阶段四：数据回流复盘

当前优先推进：

- 成片上传后回写需求、版本和制作任务状态，并进入数据回流复盘；不自动进入资产库。
- 需求详情展示版本级成片表现，包括消耗、CPI、IR、D7 ROAS、状态和建议动作。
- 方向卡片和需求大表展示数据回流信号，帮助判断放量、迭代、暂停或继续观察。
- 数据看板按周期、方向、渠道、制作类型筛选，并从方向维度展示回流结论。
- 需求脚本中的 `引用成片` 是独立参考来源，用于引用近期成片表现和复盘素材，不等同于 `引用资产库`。

### 阶段五：资产治理

后续推进：

- 资产库保持手动录入和人工治理，用于沉淀可复用资产、组件和片段。
- 资产库顶部增加类型标签筛选：左侧目录负责资产归属，顶部标签负责快速浏览和组合筛选。
- 资产库标签筛选按游戏广告视频资产组织，至少覆盖资源类型、主题题材、角色生物、玩法机制、场景环境和关键词状态。
- 资产库标签筛选区域使用纵向滚动的全量标签面板，不做大类切页；每个大类在同一面板中按分组铺开，保持浏览密度。
- 资产库主页面支持多标签组合筛选，用于同时限定资源类型、主题、玩法机制和状态；需求脚本引用弹窗可以保留更轻量的单标签筛选。
- 资产库筛选区需要展示已选标签，并支持逐个移除；暂不做常用组合入口，避免在真实标签治理前固化不稳定的运营预设。
- 资产卡片需要展示适用位置，例如 A 段、中间段、B 段、CTA、背景、UI 组件、特效、音效、角色或图片，帮助提需求时快速判断是否可用。
- 资产详情弹窗也需要展示同一套适用位置，避免用户从卡片进入详情后丢失“可用于哪类需求”的判断。
- 需求脚本里的资产引用弹窗沿用资产库同一套类型语言，避免用户在两个地方学习两套分类。
- 需求脚本里的资产引用弹窗需要根据当前上下文默认筛选：在 A 段引用时默认看 A 段资源，B 段/大字报默认看 B 段资源，图片需求默认看图片视觉，但用户可以手动切换到任意标签。
- 后续需求可以引用人工沉淀资产，并追溯资产来源说明。
