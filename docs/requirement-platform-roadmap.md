# 需求平台闭环落地路线图

## 背景

当前平台已经有需求中心、协同看板、需求详情、需求脚本、资产库、数据看板等模块雏形，但这些模块之间还没有形成稳定的业务闭环。

本路线图用于把《需求平台》需求文档拆成可开发、可验收、可逐步交付的工程计划。后续开发优先按本文推进，避免单点 UI 迭代和真实工作流脱节。

## 总目标

把创意需求从多维表维护，迁移成平台化工作流：

1. 创意创建方向。
2. 在方向下创建具体需求。
3. 需求详情自动继承方向字段并补充脚本、参考和版本信息。
4. 组长审核、拆岗、评估难度并排期。
5. 制作人员按任务推进状态并上传成品。
6. 成品关联需求、版本、尺寸和投放数据，并回流到需求与方向复盘。
7. 资产库保持手动录入和人工治理，用于沉淀可复用资产。

第一阶段目标不是做完所有模块，而是先跑通“方向 -> 需求 -> 排期 -> 制作状态”的最小闭环。

## 核心对象模型

### Direction / 创意方向

当前主要对应 `CreativeSchedule`。

它是需求的父级容器，承载周期、方向名称、验证目标、制作类型、大方向、素材阶段、优先级、创意负责人、使用场景、投放渠道、初版验收时间、截止时间、需求数量和进度。

后续建议逐步把 `CreativeSchedule` 语义收敛为 `CreativeDirection`，但短期不强行重命名，避免大范围改动。

### Requirement / 具体需求

当前对应 `Requirement`。

它是可审核、可排期、可制作、可上传成品的需求单元。需求必须通过 `scheduleId` 关联到方向。

需求从方向下新建时，应自动继承方向中的通用字段，包括：

- `assetType`
- `broadDirection`
- `materialStage`
- `creativePersonnel`
- `owner`
- `priority`
- `direction`
- `channels`
- `goal` 或验证目标

### Version / 小版本

当前主要对应 `AssetVersionItem` 和 `Requirement.subVersions`。

小版本用于承载同一大需求下的多个版本名称、验证目标、参考、描述、脚本片段和成品。

第一阶段只需要保证需求详情中能正确展示和编辑小版本；排期可以先默认整体排期，后续再支持分版本排期。

### ProductionTask / 制作任务

当前对应 `ProductionTask`。

它是排期和制作状态的最小执行单元。一个需求可以有多个任务，用于表示平面、视频、程序、3D、模型等拆岗结果。

后续应扩展字段：

- `role`：岗位或制作环节。
- `assignee`：制作人员。
- `difficulty`：难度。
- `plannedStartDate`
- `plannedEndDate`
- `actualStartDate`
- `actualEndDate`
- `estimatedWorkDays`
- `actualWorkDays`
- `lockedFields`
- `dependencyIds`

### FinishedCreative / 成片投放结果

当前新增对应 `FinishedCreativePerformance`。

成片用于承载需求版本的制作结果、投放表现、复盘结论和后续迭代建议。成片不自动进入资产库；它通过需求详情、需求大表、方向卡片和数据看板回流到复盘链路。

### Asset / 手动治理资产

当前主要对应 `LibraryItem` 和资产库组件。

资产库保持手动录入和人工治理，用于沉淀可复用资产、组件和片段。需求脚本里的 `引用资产库` 使用资产库内容；`引用成片` 使用独立成片候选池，二者不混用。

## 状态机

### 需求状态

当前 `RequirementReqStatus` 为：

- `Draft`
- `Pending`
- `Approved`
- `Modification`

建议短期保持枚举不大改，但定义清楚动作：

- `Draft`：创意创建但未提交审核。
- `Pending`：已提交，等待审核。
- `Approved`：审核通过，可以进入排期。
- `Modification`：审核退回或需补充。

关键规则：

- 未审核通过的需求不可排期。
- 审核通过后自动进入待排期状态。
- 方向卡片的需求提交进度应以 `Approved` 数量占总需求数作为主百分比。

### 制作状态

当前 `RequirementProdStatus` 为：

- `Scheduled`
- `InProgress`
- `Completed`

建议在 UI 文案中映射为：

- `Scheduled`：待排期或已排期，需要根据任务是否有人员和时间进一步显示。
- `InProgress`：制作中。
- `Completed`：已完成。

更完整的任务级状态建议沉淀在 `ProductionTask.status`：

- `待排期`
- `已排期`
- `制作中`
- `已完成`

关键规则：

- 需求审核通过后，如没有任务人员和时间，显示待排期。
- 输入人员和时间后，显示已排期。
- 制作人员点击开始制作后，任务进入制作中。
- 上传成品成功后，任务进入已完成。
- 一个需求有多个任务时，所有任务完成才算需求制作完成；任一任务开始则需求进入制作中。

## 第一阶段：需求中心闭环

### 目标

跑通协同看板、需求大表、需求详情三者的数据联动。

### 必做能力

1. 协同看板方向卡片作为方向入口。
2. 点击卡片打开该方向下所有需求详情弹窗。
3. 卡片内 `新建需求` 为最高优先级操作。
4. 在方向下新建需求时，自动填充方向已有字段。
5. 新建需求后直接打开对应制作类型的详情页。
6. 详情页根据 `assetType` 分流：
   - `Video`：视频需求脚本模板。
   - `Image`：图片需求模板。
   - `Playable`：试玩需求模板。
7. 需求大表继续保留父子关系，但字段与状态口径对齐方向和详情页。
8. 方向卡片进度由真实关联需求计算，不再依赖孤立的假数据。

### 当前已有

- 协同看板卡片、周期、筛选、排序、进度条。
- 方向下新建需求入口。
- 需求详情页基础结构。
- 视频脚本模板、图片和试玩简单模板。
- 资产库引用弹窗雏形。

### 当前缺口

- 新建需求的字段继承不完整，需要统一口径。
- 方向详情弹窗、需求大表、需求详情之间的动作还不够稳定。
- 需求状态和制作状态没有完全按状态机驱动。
- 方向卡片上的需求编号、版本预览、需求弹窗需要按文档重构。

### 第一阶段验收

- 从任意方向卡片点击 `新建需求`，可以直接进入对应类型的需求详情。
- 新需求的方向名称、制作类型、大方向、素材阶段、优先级、负责人、渠道和验证目标自动带入。
- 在需求详情修改需求状态后，回到协同看板能反映需求提交进度。
- 在需求详情修改制作状态后，回到协同看板能反映制作完成进度。
- 点击方向卡片能看到该方向下所有需求，点击某条需求能进入详情。

## 第二阶段：排期规则与制作排期

### 目标

把需求详情里的排期区做成可用的生产排期入口，并让制作排期看板消费同一份任务数据。

### 必做能力

1. 需求审核通过后才可排期。
2. 支持整体排期和分版本排期。
3. 支持岗位拆分：平面、视频、程序、3D、模型等。
4. 支持难度选择和工时输入。
5. 支持人员选择和时间选择。
6. 支持手动排期辅助：填写人员、时间和工时后，系统提示占用、撞期和截止风险。
7. 支持前后置关系：
   - 平面是视频前置。
   - 平面和视频是程序前置。
   - 地编和模型是 3D 视频前置。
8. 制作排期看板支持按人员查看日历和甘特视图。

### 后端或集成依赖

- 飞书请假同步。
- 中国节假日和调休表。
- 定时提醒和飞书消息。

### 第一版前端可先模拟

- 人员请假数据用 mock。
- 节假日先写入本地配置。
- 飞书消息用 toast 或日志面板模拟。

## 第三阶段：数据回流复盘

### 目标

让成片投放表现回流到需求、版本和方向，支持判断放量、迭代、暂停或继续观察。

### 必做能力

1. 成片投放数据关联需求编号、版本、尺寸、渠道和制作人。
2. 上传或同步成功后回写需求、版本和制作任务状态。
3. 需求详情展示版本级表现和复盘结论。
4. 需求大表展示建议动作，例如放量、迭代、暂停或继续观察。
5. 方向卡片展示聚合回流信号和风险提示。
6. 数据看板支持按方向、周期、渠道和制作类型查看复盘结果。

## 第四阶段：资产治理

### 目标

资产库保持手动录入和人工治理，沉淀可复用资产。成片可作为复盘参考，但不自动转为资产。

### 必做能力

1. 资产库支持人工录入资产、组件和片段。
2. 资产记录可填写适用位置、标签、来源说明和关联需求。
3. 需求脚本可以通过 `引用资产库` 引用人工治理资产。
4. `引用成片` 保持独立入口，用于选择近期投放成片作为参考。
5. 资产库记录引用次数和人工维护状态。

## 第五阶段：权限和通知

### 目标

让平台进入真实协作环境。

### 必做能力

1. 角色权限：创意、美宣组长、制作人员、管理员。
2. 页面权限和操作权限。
3. 注册申请审核。
4. 飞书消息：
   - 审核通过通知组长和创意。
   - 已排期通知制作人员和创意。
   - 到点未开始制作提醒。
   - 到点未完成提醒。
   - 成品上传完成通知。

## 开发优先级

### P0：必须先做

- 统一方向、需求、任务、资产的字段继承和关系。
- 协同看板新建需求 -> 详情页自动填充。
- 需求状态和制作状态驱动进度。
- 方向详情弹窗列表和详情跳转。

### P1：随后做

- 需求详情排期区。
- 整体排期和分版本排期。
- 制作排期看板读取任务数据。
- 排期规则基础配置。

### P2：再做

- 成片上传或同步后自动关联需求、版本和方向。
- 数据回流到需求、版本和方向。
- 资产库引用保持人工治理，不由成片自动沉淀。

### P3：最后接

- 飞书同步与消息。
- 权限细粒度控制。
- 后端持久化和真实接口。

## 下一步代码任务

第一批建议按以下顺序推进：

1. 在 `types.ts` 中补齐方向、需求、制作任务之间缺失的字段，尽量兼容当前结构。
2. 抽一个方向到需求的字段继承函数，统一 `handleAddRequirementForDirection` 和其他新建入口。
3. 让方向卡片点击打开“方向需求列表”弹窗，列表内点击进入需求详情。
4. 让 `新建需求` 创建后直接打开详情页，并默认进入对应制作类型模板。
5. 用真实关联需求重算方向卡片的需求提交进度和制作完成进度。

## 约束

- 不做无关大重构，优先兼容现有 Vite + React + TypeScript 结构。
- 先把前端 mock 流程跑通，再考虑真实后端接口。
- 所有状态变化必须能在协同看板、需求大表、需求详情之间同步。
- 需求脚本工作台继续遵守 `docs/requirement-script-workbench-guidelines.md`。
